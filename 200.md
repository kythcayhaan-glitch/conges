# Passage à 200+ utilisateurs

## SQLite — problème principal

SQLite ne gère qu'un seul écrivain à la fois (file lock). Avec 200 utilisateurs simultanés qui soumettent des demandes, des erreurs "database is locked" vont apparaître.

**Action requise : migrer vers MySQL ou PostgreSQL avant d'atteindre cette échelle.**

## Pagination — urgent

Plusieurs endroits chargent tout en mémoire sans limite :

- `leaveRepository->findAll()` → toutes les demandes de tous les agents
- `findByUserId()` → toutes les demandes d'un agent (pas de limite)
- `findAll()` sur les utilisateurs dans plusieurs contrôleurs (`LeaveWebController`, `DashboardController`, `AgentWebController`)

Avec 200 agents et des années de demandes, ces requêtes deviendront lentes et consommeront beaucoup de RAM.

**Action requise : ajouter la pagination (Doctrine Paginator ou KnpPaginatorBundle) sur toutes les listes.**

## PDF récapitulatif

`AgentWebController::recapPdf()` charge les 200 agents d'un coup dans dompdf. La génération sera lente.

**Action requise : envisager une génération en tâche de fond (Messenger + worker) ou limiter le périmètre du PDF.**

## Ce qui tient bien

- L'architecture DDD/CQRS est adaptée pour évoluer
- Le password hasher `auto` (bcrypt) est correct
- Les rôles et la hiérarchie de sécurité sont bien structurés

## Index de base de données

Vérifier que les colonnes suivantes sont bien indexées dans la table des demandes de congé :

- `user_id`
- `statut`
- `type`

Sans ces index, les requêtes filtrées deviendront très lentes sur un grand volume.

## Priorité suggérée

1. Migrer vers MySQL / PostgreSQL
2. Ajouter la pagination sur les listes de demandes et d'agents
3. Revoir la génération PDF (tâche de fond ou pagination)
4. Vérifier / ajouter les index
